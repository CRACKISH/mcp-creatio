🧵 Червона нитка

Усе сходиться до порожнього Engine. Із другим бекендом він стає несучим: readonly + audit + нормалізація мусять жити НАД контрактом, інакше дублюються в обох провайдерах. Те, що було YAGNI, стає стіною.

02

Контракт запиту: filter · sort · select · expand · paging · count

Нейтральним стає увесь read-запит, не лише фільтр. Тул приймає типізований ReadQuery, провайдер білдить його повністю у свою мову, а різні відповіді нормалізуються назад у спільну форму { items, totalCount? }.

вхід тулу · Zod ReadQuery
ReadQuery (наші типи)
{ select:["Name","Owner.Name"], filter:{…}, sort:[{field:"CreatedOn",dir:"desc"}],
  expand:["Owner"], top:50, skip:0, count:true }
↓   Engine → активний провайдер → QueryTranslator   ↓

→ OData

query string
$select=Name&$expand=Owner($select=Name)
&$orderby=CreatedOn desc
&$top=50&$count=true

→ DataService

request JSON
{ columns:[{path:"Name"},
   {path:"Owner.Name"}], filters:{…},
  rowCount:50, isPageable:true }
можливістьOData білдDataService білд
select$selectcolumns[].path
filter$filter рядок + Xxx/Id, bare-GUIDfilters дерево
сортування$orderby Name descorderDirection + orderPosition
paging$top / $skiprowCount / isPageable
count$count → @odata.countпрапор rowCount → лічильник
expand ⚠️$expand=Nav($select) — навігація"Nav.Field" у columns — НЕ expand
⚠️ Чому контракт, а не passthrough: expand у бекендах принципово різний — OData розкриває навігацію окремою конструкцією, DataService адресує вкладене полем-шляхом. Прокинути OData-параметри наскрізь у DataService неможливо. Тому нейтральний ReadQuery + нормалізація відповіді (value/@odata.count vs rows/rowCount → {items,totalCount}) обов'язкові. OData-квірки (XxxId→Xxx/Id, bare-GUID) переїжджають у OData-транслятор, де їм і місце.
